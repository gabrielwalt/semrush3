description: Decide WHICH pages of a source site to import — separate hand-authored marketing/brand content (the default scope) from database-backed/programmatic content (excluded) and documentation (deferred, opt-in). Use when scoping a new site, when site discovery returns far more URLs than a team could hand-author, when one URL group dwarfs all others, or whenever choosing what to analyze/import. Extends EXCAT `excat-site-scope`, `excat-url-discovery`.
---

Import only **hand-authored brand/marketing pages** by default. Programmatic (database-backed) and documentation content are out of the default scope. Triage every site's URL groups into three buckets before analyzing or importing — classify by *how a page is produced and maintained*, not by its path name.

## The Content-Triage Rule (gate — do this BEFORE sampling/analysis)

Classify each URL group into exactly one bucket using the signals below. First match wins:

| Bucket | Defining signal (how it's produced/maintained) | Action |
|--------|------------------------------------------------|--------|
| **Programmatic / database-backed** | Pages generated by stamping ONE template over rows of a dataset. Tells: a single group is a large fraction of all URLs; near-uniform path depth; path segments are data values (IDs, slugs, geos, categories) rather than editorial names; count far exceeds what a marketing team could hand-write. | **EXCLUDE.** It's a database rendered as pages, not authored content — re-generated from the data source, not migrated. |
| **Documentation** | Reference/support material kept in sync with a product: help center, knowledge base, API/developer docs, support articles, tutorials, learning courses. Often a large, separately-maintained corpus with its own templates and a different fidelity bar. | **DEFER.** Out of default scope. Import brand/marketing first; pull docs in later only if the user explicitly extends scope. |
| **Brand / marketing** | Individually hand-authored, editorially-owned pages that express the brand: home, product/solution, pricing, customers, about, editorial posts, campaign/landing pages, resources. Bounded count; each page is unique. | **INCLUDE.** This is the default import scope. |

When a group's bucket is genuinely ambiguous, decide by first match: (1) explicit user instruction; (2) is it re-generated from a dataset? → programmatic; (3) is it product-reference material? → documentation; (4) else → marketing.

## Recipe

1. After URL grouping, print the group-size distribution (largest first) and each group's path-depth uniformity.
2. Apply the triage table. A single group that is a large fraction of all URLs with uniform depth and data-like path segments is the programmatic tell — exclude it WITHOUT per-page analysis.
3. Confirm the split with assert-then-confirm: *"Group X looks database-backed and Y looks like documentation, so I'll scope the import to the marketing groups first — confirm?"* Ask only about genuinely ambiguous groups; don't dump a menu.
4. Analyze/sample only the marketing groups. Record excluded/deferred groups in the scope report so the decision is traceable.
5. If the user later extends scope ("now do the docs"), re-include that bucket as a separate import phase.

## Pitfalls
- Sampling tooling caps at N URLs and asks you to raise the cap → don't. A huge required cap is the symptom of an un-triaged programmatic section; exclude the section instead of sampling it.
- Classifying by path name alone → a marketing-looking path can be programmatic and vice-versa. Judge by production model (dataset-stamped vs hand-authored), not the URL string.
- Treating documentation as marketing → docs carry their own templates and fidelity bar; keep them a separate phase.
- Importing one example of a programmatic template "to be safe" → still out of scope unless the user asks; it pollutes the block/template catalog with a pattern we won't maintain.
